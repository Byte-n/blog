在将飞书文档转换为 Markdown 的场景中，文本、表格、图片等常规块类型的处理在之前以及完成。但在日常接触的业务需求中，有一部分产品的业务流程图是放在飞书的需求文档的画板中，在此之前都依赖截图或人工转述的方式投喂给 Agent 。
0.2.0 版本的核心目标，正是通过架构层面的重构，为画板这类高复杂度内容提供可扩展、多格式输出的渲染能力。

## 二、架构重构：从单文件解析到三阶段 AST 管线

旧版本采用单文件解析器直接输出 Markdown 字符串，这种方式在新增块类型或处理异步资源（如图片下载、画板接口请求）时，耦合严重、难以扩展。每次新增一种块类型，都需要修改核心解析逻辑，且同步和异步处理混在一起，代码可维护性较差。

0.2.0 版本将转换流程拆分为三个独立阶段：

```
┌─────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 飞书 API │───▶│   Parser    │───▶│ Transformer │───▶│  Serializer │
│ 原始数据 │    │  (AST 构建)  │    │  (异步处理)  │    │ (Markdown)  │
│          │    │  同步、纯函数 │    │  网络请求    │    │  字符串输出  │
└─────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

**Parser 阶段**负责将飞书返回的扁平化 Block 数组转换为嵌套的 AST 节点树。每种 Block 类型对应一个独立的解析器，通过注册表机制动态挂载。这里采用注册表模式（Registry Pattern）而非传统的 switch-case，是因为注册表支持运行时动态注册新解析器，符合开闭原则——新增块类型只需添加新文件并注册，无需修改已有代码。解析器只关注结构映射，不涉及任何外部 IO。

**Transformer 阶段**负责异步后处理。此时 AST 已构建完成，可以一次性收集所有需要外部解析的资源 token（如图片 token、画板 token），批量调用飞书接口，然后将解析结果回填到 AST 中。批量请求的策略是：以 5 个 token 为一组并发请求，既避免单次请求过多导致超时，又显著减少了网络往返次数。这种"收集→批量请求→替换"的模式，在处理包含数十张图片的文档时效果尤为明显。

**Serializer 阶段**负责将处理后的 AST 序列化为最终的 Markdown 字符串。同样是注册表驱动的设计，每个 AST 节点类型对应独立的序列化器。

三阶段分离后，新增一种块类型的成本被控制在"实现解析器 + 实现序列化器"两个步骤内，且不会干扰现有逻辑。AST 节点采用联合类型（Union Type）定义，包含 `MdBlockNode`（块级节点）和 `MdInlineNode`（行内节点）两种结构，确保类型安全。

## 三、画板功能概览

本版本完整支持飞书画板内容的渲染，提供四种输出格式以适应不同场景：

| 格式 | 适用场景 |
|------|----------|
| `svg`（默认） | 独立文件 + Markdown 相对路径引用，适合常规文档存档 |
| `inline-svg` | SVG 标签直嵌 Markdown，适合平台支持内嵌 SVG 的场景 |
| `base64` | Data URI 内嵌，适合单文件分发、无需依赖的场景 |
| `yaml` | 结构化代码块，适合 AI 理解、程序化处理的场景 |

支持的画板节点类型涵盖组合图形、连接线、表格、思维导图、文本框、图片、分组容器、便签、区域框、画笔涂鸦、内嵌 SVG 等十余种。其中组合图形子类型支持矩形、圆角矩形、椭圆、菱形、三角形、五边形、六边形、八边形、星形、平行四边形等二十余种。

## 四、画板渲染的核心实现

画板渲染的本质是将飞书返回的扁平化节点数组转换为可视化的 SVG。其核心挑战在于：节点之间不存在显式的父子层级（除思维导图外），位置信息是绝对坐标，且连接线需要动态计算端点。

### 4.1 注册表驱动的节点渲染器

画板渲染采用与 AST 管线同构的注册表架构。每种节点类型实现一个渲染器接口，通过统一的注册中心管理：

```
interface NodeRenderer {
  type: string                    // 节点类型标识，如 'composite_shape'
  render(node, context): string   // 返回 SVG 片段
}
```

注册表模式的优势在于：画板节点类型多达 10+ 种，如果使用 if-else 或 switch-case 判断，会导致单个文件膨胀到数百行，且违反单一职责原则。通过注册表，每种节点类型的渲染逻辑独立成文件，新增节点类型只需实现接口并注册即可。渲染器不直接操作全局状态，而是通过 `RenderContext` 读取预计算的共享数据（如节点 ID 映射、思维导图父子关系、被引用节点集合等），保证纯函数特性，便于测试和组合。

### 4.2 节点上下文预处理

在正式渲染前，系统会对原始节点数组进行一次预处理，构建渲染管线共享的上下文：

- **nodeMap**：以节点 ID 为键的快速查找表，用于连接线端点解析、子节点定位。
- **mindMapChildrenMap**：思维导图的父→子映射表。飞书返回的思维导图节点通过 `parent_id` 隐式关联，需提前建立索引以便递归渲染。
- **rootNodes**：过滤出真正需要渲染的根节点。子节点（如分组内的成员、思维导图的子节点）会通过递归或引用机制渲染，避免重复。
- **referencedIds**：记录被连接线或思维导图关联引用的节点，用于后续渲染属性标注。

### 4.3 自适应视口计算

SVG 需要明确的 `viewBox` 范围（即画布的可见区域，格式为 `minX minY width height`）。系统通过遍历所有根节点，计算整体包围盒：

- 对于普通节点，取其坐标与宽高。
- 对于旋转节点，通过四个角点旋转变换后重新计算极值，确保包围盒精确。
- 对于连接线，解析其起点、终点及所有转折点的实际坐标后纳入计算。
- 对于画笔涂鸦，逐点展开后取极值。

最终包围盒会附加一定边距（Padding），保证渲染内容不会贴边，视觉效果更舒适。

### 4.4 组合图形：路径函数映射

组合图形的子类型多达二十余种，但渲染逻辑可以抽象为统一的"路径生成"模式：首先根据形状类型查找对应的路径生成函数，传入宽高和样式参数得到 SVG 路径描述字符串，然后叠加文本内容渲染和子节点渲染结果，最终返回完整的 `<g>` 分组元素。未识别的形状类型自动降级为圆角矩形，保证健壮性。

### 4.5 连接线：端点解析与路径构建

连接线的渲染涉及两类复杂度：端点定位和路径形状。

**端点解析**支持三种语义：
- 附着到某节点的相对位置（如"连接至矩形右侧中点"）
- 绝对坐标
- 兼容旧版数据格式的对象引用

解析时会查询 nodeMap 将相对位置转换为绝对像素坐标。

**路径构建**支持三种线型：
- 直线：直接连接各点
- 折线：按转折点依次连接
- 曲线：使用贝塞尔曲线或二次曲线进行平滑插值

此外，连接线支持起止箭头样式（通过 SVG `marker` 定义）、虚线样式、边框宽度、以及中点文本标注。

### 4.6 思维导图：递归渲染与方向推断

思维导图的节点在数据层面是扁平的，通过 `parent_id` 隐式关联。渲染时从根节点出发，递归渲染子节点：首先从父子映射表中获取当前节点的子节点列表，对每个子节点推断其相对方向（右/左/上/下），计算父子之间的连线路径（支持直角、圆角、曲线三种线型），以局部坐标系渲染子节点形状和文本，递归处理孙节点，最后将所有内容组装为 `<g>` 分组。

方向推断优先读取根节点记录的方向列表；若不存在，则通过比较父子节点的几何中心自动推断。所有子节点渲染在父节点的局部坐标系内，通过 `transform="translate(...)"` 实现定位，保证嵌套层级清晰。

### 4.7 YAML 管线：为 AI 场景设计的结构化输出

除视觉渲染外，画板还支持输出为结构化 YAML。该管线独立于 SVG 渲染，但复用同一套节点上下文预处理：执行节点预处理后，将每个根节点转换为中间 JSON 结构，递归处理子节点（思维导图、分组等），收集所有图片 token 用于后续替换，最终序列化为 YAML 字符串。

YAML 结构保留了画板的语义信息（节点类型、位置、尺寸、文本内容、样式、层级关系），但剔除了 SVG 的路径细节，更适合大模型理解。图片支持在线链接和本地路径两种模式。这种双管线设计（SVG + YAML）体现了"一份数据、多份输出"的理念：视觉渲染面向人类读者，结构化输出面向 AI 和程序处理。

## 五、工程化改进

### 5.1 Converter 公共 API

新增 `converter.ts` 导出 `convert()` 函数，支持作为库编程调用。npm 包入口由 CLI 脚本改为 Converter 模块，使其同时具备命令行工具和可编程库的双重身份。

### 5.2 CLI 增强与环境变量

画板相关的 CLI 参数包括：
- `--wb-format`：选择画板输出格式
- `--wb-bg`：SVG 背景样式（纯色/点阵/透明）
- `--wb-image-mode`：画板内嵌图片的处理方式（本地下载 / base64 / 在线链接）

所有 CLI 参数均支持通过环境变量配置（如 `LARK_DOCX2MD_WB_FORMAT`），便于 CI/CD 集成。

### 5.3 Agent 模式联动

新增 `--agent` 标志，开启时自动强制：
- `image-mode=online`（避免本地文件依赖）
* `wb-image-mode=online`（避免本地文件依赖）
- `wb-format=yaml`（输出 AI 友好的结构化数据）

这一设计使得工具在 Agent/大模型工作流中可以直接嵌入，无需额外参数调优。

## 六、使用示例

### CLI 使用

```bash
# 基础用法（默认输出 SVG 格式）
lark-docx2md --url <飞书文档链接>

# 指定画板输出为 YAML 格式
lark-docx2md --url <飞书文档链接> --wb-format yaml

# Agent 模式（自动强制 image-mode=online 和 wb-format=yaml）
lark-docx2md --url <飞书文档链接> --agent
```

### 环境变量配置

```bash
# 通过环境变量配置凭证和参数
export LARK_DOCX2MD_APP_ID=your_app_id
export LARK_DOCX2MD_APP_SECRET=your_app_secret
export LARK_DOCX2MD_WB_FORMAT=yaml

lark-docx2md --url <飞书文档链接>
```

### 作为库调用

```typescript
import { convert } from 'lark-docx2md';

const result = await convert({
  url: 'https://your-domain.feishu.cn/docs/xxx',
  appId: 'your_app_id',
  appSecret: 'your_app_secret',
  output: './output',
  wbFormat: 'yaml',
  agent: true,
});

console.log(result.markdown);  // Markdown 内容
console.log(result.docToken);  // 文档 Token
```

## 七、结语

0.2.0 版本的改动不仅仅是"新增了画板支持"，更是通过 AST 三阶段管线的架构升级，为后续更多复杂块类型的处理建立了可扩展的范式。画板渲染本身也采用了与 AST 管线同构的注册表+上下文驱动设计，保证了两层架构在理念上的一致。

从工程角度看，SVG 渲染管线保证了视觉保真度，YAML 管线则开辟了 AI 语义理解的路径，两者共享底层基础设施却面向不同场景——这也是本次设计中"一份数据、多份输出"思路的体现。
